# Laptop Migration Guide

## Overview

This directory contains scripts to export and import config/credential files when migrating to a new macOS laptop. The migration tarball (`migration-configs.tar.gz`) and a Homebrew `Brewfile` are generated on the source machine and applied on the target.

---

## Instructions

### On the old laptop

```bash
./config/migrate-export.sh
```

This creates `~/Desktop/laptop-migration/` containing:
- `migration-configs.tar.gz` â€” all config and credential files listed below
- `Brewfile` â€” snapshot of installed Homebrew formulas and casks

Transfer the `laptop-migration/` folder to the new laptop (e.g. AirDrop, USB drive, or encrypted transfer).

### On the new laptop

1. Clone this repo and run the base installer first:
   ```bash
   ./install/install.sh
   ```
2. Place the `laptop-migration/` folder on the Desktop (or anywhere accessible).
3. Run the import script:
   ```bash
   ./config/migrate-import.sh              # reads from ~/Desktop/laptop-migration/
   ./config/migrate-import.sh /path/to/dir  # or specify a custom location
   ```
4. Review the final report printed by the script.

## Contents of the Migration Archive

### 1. Credentials and Secrets

| Item | Source Path | Destination | Notes |
|------|------------|-------------|-------|
| SSH keys and config | `~/.ssh/` | `~/.ssh/` | ED25519 keypair, host config, known_hosts. `agent/` socket excluded. Permissions fixed to 700/600 on import. |
| AWS config | `~/.aws/` | `~/.aws/` | Region config. Credentials may use SSO/IAM role instead of static keys. |
| Kubernetes config | `~/.kube/config` | `~/.kube/config` | Cluster certs and contexts for EKS clusters. Cache dir excluded. |
| GitHub CLI auth | `~/.config/gh/` | `~/.config/gh/` | OAuth token for git/gitsoma. **Consider rotating token after migration.** |
| Docker config | `~/.docker/config.json` | `~/.docker/config.json` | Registry auth entries. Actual passwords stored in macOS Keychain via `credsStore: desktop`. |
| Docker daemon | `~/.docker/daemon.json` | `~/.docker/daemon.json` | Daemon settings (GC, experimental flags). |

### 2. Shell and Git Config

| Item | Source Path | Destination | Notes |
|------|------------|-------------|-------|
| Git config | `~/.gitconfig` | `~/.gitconfig` | User identity, Git LFS filter. Also recreatable via `install/setup-git.sh`. |
| Zsh config | `~/.zshrc` | `~/.zshrc` | Oh My Zsh setup. Also tracked at `config/.zshrc` in this repo. |
| Zsh history | `~/.zsh_history` | `~/.zsh_history` | Command history. |
| z database | `~/.z` | `~/.z` | Frecency data for the `z` directory jumper. |

### 3. Dev Tool Configs

| Item | Source Path | Destination | Notes |
|------|------------|-------------|-------|
| Cursor settings | `~/Library/Application Support/Cursor/User/settings.json` | Same path | Editor preferences. Profile also tracked at `config/default_cursor_profile.code-profile`. |
| Cursor keybindings | `~/Library/Application Support/Cursor/User/keybindings.json` | Same path | Custom key bindings. |
| iTerm2 preferences | `~/Library/Preferences/com.googlecode.iterm2.plist` | Same path | Full iTerm2 config. Export also at `config/iTerm2 State.itermexport`. |

### 4. Browser Profile (Chrome)

| Item | Source Path | Destination | Notes |
|------|------------|-------------|-------|
| Chrome profile | `~/Library/Application Support/Google/Chrome/` | Same path | Cookies, local storage, IndexedDB, history, bookmarks, saved passwords, extensions, autofill, preferences. Bulk cache dirs (`Cache`, `Code Cache`, `GPUCache`, `blob_storage`, `Session Storage`, etc.) are excluded â€” they regenerate automatically. **Chrome must be quit during import.** |

### 5. Homebrew Packages

| Item | File | Notes |
|------|------|-------|
| Brewfile | `Brewfile` (alongside tarball) | Generated by `brew bundle dump`. Applied via `brew bundle`. |

## Agent Instructions

> **For AI agents applying this migration:** Each config item MUST be applied independently. Do not bail out on the first failure â€” continue applying all remaining items. At the end, produce a final report listing which items were applied successfully and which failed (with the reason). The `migrate-import.sh` script already does this; if applying manually, follow the same pattern.

Key rules:
- Back up any existing file at the destination before overwriting.
- Fix SSH key permissions (`chmod 700 ~/.ssh`, `chmod 600 ~/.ssh/id_ed25519`) after copying.
- For directory items, merge contents rather than replacing the entire directory.
- `brew bundle` failures are non-fatal â€” some casks may need manual installation.
- Report results in a structured format: item name, status (success/failed), and error detail if failed.
